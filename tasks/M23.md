# M23 — AI-Augmented Planning

> **Status:** DESIGN PHASE
> **Depends on:** M22 (Intention-Driven Autonomous Agent) — COMPLETE
> **Frozen milestones:** M17 (nova_screen), M18 (nova_input), M19 (Task Execution), M20 (Planning + World State), M21 (Closed-Loop Execution), M22 (Intention-Driven Autonomous Agent)
> **Location:** `modules/automation/src/planner.rs` (additive changes) + minor wiring in `modules/automation/src/ai_bridge.rs`
> **Key constraint:** No changes to any frozen public API. No breaking changes. No new crates.

---

## Objective

Connect nova_ai to the Planner so that NOVA can decompose **any natural language goal** into an executable `ExecutionPlan`, not just the 14 predefined heuristic patterns. This is the single biggest capability gap after M22: the Planner's `plan()` fallback creates a no-op `ActionType::RunAI` that simply logs the prompt and returns success — it never invokes an AI, never generates real steps, and never produces a meaningful plan for novel goals.

M23 transforms the Planner from a brittle pattern-matcher into an **AI-augmented decomposer** that uses the existing nova_ai engine to break down novel goals into validated, executable step sequences while keeping heuristic patterns as fast-paths for known operations.

## Functional Requirements

### FR-M23-001 — AI-Powered Goal Decomposition
The system MUST be able to decompose any natural language goal description into an `ExecutionPlan` using the AI engine. The AI MUST receive:
- The complete set of available `ActionType` variants with descriptions
- The `Capability` enum values with descriptions
- The `ExecutionStep` schema (id, description, action, dependencies, required_capabilities, timeout_ms, retry_count, continue_on_failure)
- The `ExecutionPlan` schema
- The goal description and context
- Examples of well-formed plans for known goal types

The AI MUST return a structured JSON plan that:
- Contains 1–20 steps (configurable via `max_steps_per_plan`)
- Uses only valid `ActionType` and `Capability` values
- Has no circular dependencies
- Has concrete, actionable step descriptions

### FR-M23-002 — Heuristic Fast-Path Preservation
The existing 14 heuristic decomposition patterns MUST remain as fast-path for known goal types. AI planning MUST only be invoked when heuristic decomposition would fall through to the `RunAI` fallback. This ensures:
- Known patterns like "open calculator" still resolve instantly (< 10 ms)
- AI planning is only invoked for novel/unseen goals
- No regression in existing planner behavior

### FR-M23-003 — Plan Validation After AI Generation
Every AI-generated plan MUST pass through `Planner::validate()` before being accepted. If validation fails, the system MUST:
- Return specific validation errors to the AI (e.g., "step 's1' has unknown action type 'Foo'")
- Allow the AI to retry with corrected output (up to configurable max retries)
- If max retries exceeded, fall back to a graceful error explaining the goal cannot be planned

### FR-M23-004 — Graceful AI Unavailability Handling
If the AI engine is unavailable, returns invalid output repeatedly, or times out, the system MUST:
- Return an error explaining the goal cannot be decomposed
- Suggest rephrasing or providing more specific instructions
- NOT fall back to the no-op `RunAI` action (which would produce a misleading success)

### FR-M23-005 — Context-Aware Planning
The AI planner MUST receive goal context (`HashMap<String, String>`) and use it to produce more specific plans. For example:
- `Goal { description: "find files", context: {"location": "/home/user/docs"} }` → plan uses the specified location
- `Goal { description: "send message", context: {"contact": "alice"} }` → plan targets the specified contact

### FR-M23-006 — Pipeline Integration
The AI planner MUST be wired into the execution pipeline so that when `AiAutomationBridge::process_request()` receives a goal that the heuristic planner can't decompose, it invokes `plan_with_ai()` before falling back to error.

#### AI Planning Flow

```
plan_with_ai(goal)
  │
  ├─→ build_planning_prompt()
  │     Describes: ActionType variants, Capability enum,
  │     ExecutionStep schema, ExecutionPlan schema,
  │     examples for 2-3 goal types, validation rules
  │
  ├─→ ai.complete_structured(system_prompt, goal_json)
  │     │
  │     ├─→ Success → parse JSON → validate
  │     │     ├─→ Valid → return AiPlanResult::Plan(plan)
  │     │     └─→ Invalid → collect errors → retry
  │     │
  │     ├─→ Clarification → return AiPlanResult::Clarification
  │     └─→ Error → retry or return AiPlanResult::Failed
  │
  └─→ After max retries → return AiPlanResult::Failed
```

#### System Prompt Design

The system prompt must be concise (< 4 KB target, 8 KB hard limit) and include:

1. **Role description:** "You are an AI planning assistant that decomposes user goals into executable step sequences."
2. **Action types catalogue:** All `ActionType` variants with concise descriptions
3. **Capability catalogue:** All `Capability` variants with descriptions
4. **Step schema:** JSON schema for `ExecutionStep`
5. **Plan schema:** JSON schema for `ExecutionPlan`
6. **Validation rules:**
   - Return valid JSON matching the schema exactly
   - Use only the ActionType and Capability values listed above
   - Step IDs must be unique, sequential ("s1", "s2", ...)
   - Dependencies must reference existing step IDs
   - No circular dependencies
   - Each step must have a clear, actionable description
7. **Examples:** 2-3 minimal examples (e.g., a 1-step open-app plan, a 2-step search plan)
8. **Error handling:** If the goal is ambiguous or unclear, return a clarification request instead of a plan

#### Heuristic/AI Routing in `plan()`

```rust
impl Planner {
    pub fn plan(&self, goal: &Goal) -> Result<ExecutionPlan, String> {
        let description = goal.description.trim().to_lowercase();
        let steps = self.decompose(&description, goal)?;

        if steps.is_empty() {
            return Err("could not decompose goal".into());
        }

        // Check if the result is the RunAI fallback (single step, AiInference capability)
        if self.is_runai_fallback(&steps) {
            // Heuristic couldn't handle it — need AI planning
            // But plan() is sync, so we return an error that triggers AI path upstream
            return Err("needs_ai_planning".to_string());
        }

        // Normal heuristic path — build plan as before
        ...
    }

    /// Detect whether the decompose result is the RunAI fallback.
    fn is_runai_fallback(&self, steps: &[ExecutionStep]) -> bool {
        steps.len() == 1
            && matches!(steps[0].action, ActionType::RunAI { .. })
            && steps[0].required_capabilities == vec![Capability::AiInference]
    }
}
```

Wait — this changes the return type semantics of `plan()`. Let me reconsider.

Actually, I want to avoid changing the existing `plan()` signature or behavior. The heuristic path should work exactly as before. Instead, I'll add a separate method and the wiring happens at the caller level (`ai_bridge.rs`).

Better approach: Keep `plan()` as-is (it returns `ActionType::RunAI` for novel goals, same as before). Add `plan_with_ai()` as a new method. The caller in `ai_bridge.rs` checks if the heuristic plan is a `RunAI` fallback and calls `plan_with_ai()` instead.

```rust
impl Planner {
    /// Existing heuristic planning — UNCHANGED from M22.
    pub fn plan(&self, goal: &Goal) -> Result<ExecutionPlan, String> { ... }

    /// NEW: AI-powered planning for novel goals.
    /// Called when the heuristic plan is a RunAI fallback.
    pub async fn plan_with_ai(&self, goal: &Goal) -> Result<AiPlanResult, String> { ... }

    /// NEW: Check if a plan is the RunAI fallback (heuristic couldn't handle it).
    pub fn is_runai_plan(&self, plan: &ExecutionPlan) -> bool { ... }
}
```

This is clean, non-breaking, and follows the pattern of additive changes.

### Subsystem 2 — Pipeline Wiring

**File:** `ai_bridge.rs` (additive changes)

**Purpose:** When the heuristic planner returns a `RunAI` fallback plan, route through `plan_with_ai()` instead.

#### Modified Flow in `process_request()`

```
Current M22 flow:
  IntentParser::parse(text) → Goal → Planner::plan(goal) → ExecutionPlan
    └─ If plan uses RunAI → still proceeds with no-op plan

New M23 flow:
  IntentParser::parse(text) → Goal → Planner::plan(goal) → ExecutionPlan
    └─ If plan uses RunAI (heuristic fallback):
         ├─→ Planner::plan_with_ai(goal) → AiPlanResult
         │     ├─→ Plan → use AI-generated plan
         │     ├─→ Clarification → return Clarify decision to user
         │     └─→ Failed → return Error decision to user
         └─ Else use heuristic plan (fast-path)
```

#### Code Changes in `ai_bridge.rs`

```rust
// In process_request(), after plan = self.planner.plan(&goal):
let plan = match self.planner.plan(&goal) {
    Ok(p) if self.planner.is_runai_plan(&p) => {
        // Heuristic couldn't handle this — try AI planning
        match self.planner.plan_with_ai(&goal).await {
            Ok(AiPlanResult::Plan(ai_plan)) => ai_plan,
            Ok(AiPlanResult::Clarification { question }) => {
                return Ok(AutomationResponse::success(
                    AutomationDecision::Clarify {
                        question,
                        options: vec![],
                    },
                    "I need clarification to plan this goal",
                ));
            }
            Ok(AiPlanResult::Failed { reason }) | Err(reason) => {
                return Ok(AutomationResponse::error(format!(
                    "Could not plan this goal: {}. Try rephrasing or being more specific.",
                    reason
                )));
            }
        }
    }
    Ok(p) => p,  // Heuristic plan was sufficient
    Err(e) => {
        return Ok(AutomationResponse::error(format!("planning failed: {}", e)));
    }
};
```

## Production-Readiness Criteria

1. **No regressions:** All 528+ existing automation tests pass, all workspace tests pass
2. **No no-op RunAI paths:** After M23, `ActionType::RunAI` is never used as a planning fallback (it may still be used for legitimate AI inference steps within plans)

5. **Observability:** AI planning attempts, retries, and outcomes are observable (via existing event bus or logging)
6. **Configurable:** `Planner::with_max_ai_retries()` and `Planner::with_ai()` are configurable, no hardcoded AI dependencies
