# M22 — Intention-Driven Autonomous Agent

> **Status:** DESIGN — NOT IMPLEMENTED
> **Depends on:** M21 (Closed-Loop Autonomous Execution) — COMPLETE
> **Frozen milestones:** M17 (nova_screen), M18 (nova_input), M19 (Task Execution), M20 (Planning + World State), M21 (Closed-Loop Execution)
> **Location:** `modules/automation/src/` (new files) + minor additions to existing files
> **Key constraint:** No changes to any frozen public API. No breaking changes. No new crates.

---

## Objective

Connect nova_ai's language understanding with nova_automation's execution pipeline so that users can express goals in **natural language** and have NOVA understand, plan, execute, explain, and remember them. Add persistence so execution history survives restarts, and a bidirectional bridge so the AI can autonomously trigger and inspect automation during conversation.

This transforms NOVA from a system that can *only* execute programmatic `Goal` structs into one that **understands what you want and acts on it** — the core of an autonomous agent.

## Why M22 Logically Follows M21

| Milestone | What it built | Role |
|-----------|---------------|------|
| M12 | Automation engine, workflows, triggers, scheduler | Automate |
| M17 | Screen capture, OCR, UI tree, visual grounding | **Sense** |
| M18 | Mouse/keyboard/touch simulation | **Act** |
| M19 | ComputerController, real executors, consent gate | **Act with recovery** |
| M20 S1 | Planner: goal decomposition, topological sort, plans | **Plan** |
| M20 S2 | WorldState: device telemetry, diffs, subscriptions | **Context** |
| M21 | PlanExecutor, OutcomeVerifier, RecoveryOrchestrator, Observability | **Closed-loop execute** |
| **M22** | Intention Parser, Goal Registry, Execution Manager, AI Automation Bridge, Feedback Generator | **Understand & manage** |

**M22 is the intelligence layer.** M12–M21 built the mechanical systems. M22 gives them understanding, memory, and a conversational interface.

The key insight: M21's pipeline works perfectly when given a `Goal` struct, but `Goal` has only a `description: String` field that the Planner decomposes via brittle heuristic keyword matching. The AI engine (nova_ai) exists but has no connection to automation. M22 bridges this gap.

| Before M22 | After M22 |
|------------|-----------|
| `Goal::new("set brightness to 50")` works via keyword matching | `"Hey Nova, please dim the screen a bit"` → AI understands *intent* → resolves to `Goal` |
| Planner uses 14 hardcoded keyword patterns | Planner uses AI for novel/unseen goals, falls back to heuristics for known patterns |
| Execution history is ephemeral (RAM only) | Goals, plans, and reports persist in SQLite across restarts |
| No way to queue or manage multiple goals | ExecutionManager with submit/status/cancel/resume lifecycle |
| AI can't trigger automation during chat | AI tools: `execute_goal`, `check_status`, `list_history`, `cancel` |
| Results are raw structs, not user-friendly | FeedbackGenerator converts reports to natural language summaries |

## Functional Requirements

### FR-M22-001 — Natural Language Goal Resolution
The system MUST accept natural language text and produce a validated `Goal` struct using AI understanding. The resolver MUST handle:
- Simple goals: *"dim the screen"* → `Goal { description: "set brightness to 50%", ... }`
- Parameterized goals: *"set brightness to 75%"* → extracts `75` as the value
- Compound goals: *"open Chrome and search for Rust"* → multi-step plan with dependencies
- Ambiguous goals: *"make it darker"* → clarify as brightness vs. dark mode
- Unknown goals: *"show me my photos from yesterday"* → attempt AI decomposition

### FR-M22-002 — Goal Execution Lifecycle
The system MUST manage goals through a lifecycle: `Submitted → Resolving → Resolved → Planned → Executing → Completed / Failed / Cancelled`. Each transition MUST be persisted and queryable.

### FR-M22-003 — Execution Persistence
Goals, execution plans, step outcomes, and execution reports MUST be persisted to SQLite and survive application restarts. The persistence layer MUST support:
- Save goal with full context
- Save/load execution plan
- Save/load execution report with all step results
- Query goals by status, time range, and text search
- Purge old records (configurable retention)

### FR-M22-004 — AI Automation Tools
The AI engine MUST be able to trigger and inspect automation through tools:
- `execute_goal(text)` — submit a natural language goal for execution
- `check_goal_status(execution_id)` — get current status and progress
- `list_goal_history(limit, status_filter)` — browse past executions
- `cancel_execution(execution_id)` — cancel a running goal
- Each tool MUST pass through consent gate and log to activity trail

### FR-M22-005 — Feedback Generation
After a goal completes, the system MUST produce a natural language summary:
- What was requested
- What steps were taken
- Which steps succeeded/failed/skipped
- How long it took
- Any errors or escalations
- The summary MUST be published as an event and available via the execution report

### FR-M22-006 — Multi-Goal Queue
The system MUST support submitting goals while another is executing. Subsequent goals MUST be queued and executed sequentially. The queue MUST be inspectable (view pending goals, reorder, cancel queued goals).

### FR-M22-007 — Consent Gate Integration via AI
When the AI resolves a goal, the system MUST pre-classify the entire plan (reuse M19 `ActionClassifier`) and present a summary consent prompt before execution begins. The AI MUST NOT bypass consent — every execution still goes through the consent gate.

### FR-M22-008 — Restart Resilience
If the application restarts while a goal is executing:
- The execution MUST be marked as `Interrupted` in persistence
- The plan and completed steps MUST be preserved
- The user MAY resume from the interruption point

## Subsystem Breakdown

### Subsystem 1 — Intention Parser

**File:** `intention_parser.rs`

**Purpose:** Convert natural language text into structured `Goal` objects using AI understanding, with heuristic fallback.

```rust
/// Result of resolving a natural language intention.
pub enum IntentionResult {
    Goal(Goal),                           // Fully resolved, ready to plan
    Goals(Vec<Goal>),                     // Compound: multiple independent goals
    ClarificationNeeded {                 // Ambiguous — need user input
        question: String,
        alternatives: Vec<String>,
    },
    Unsupported(String),                  // Cannot be handled
}

/// Resolves natural language text into actionable goals.
pub struct IntentionParser {
    ai: Option<Arc<AIEngine>>,
    planner: Arc<Planner>,
}

impl IntentionParser {
    pub fn new(planner: Arc<Planner>) -> Self;

    /// Set the AI engine for NL understanding.
    pub fn with_ai(mut self, ai: Arc<AIEngine>) -> Self;

    /// Resolve a natural language request into a goal.
    /// Uses AI if available, falls back to heuristic keyword matching.
    pub async fn resolve(
        &self,
        text: &str,
        context: Option<&HashMap<String, String>>,
    ) -> IntentionResult;

    /// Resolve using AI only.
    async fn resolve_with_ai(
        &self,
        text: &str,
        context: Option<&HashMap<String, String>>,
    ) -> Result<IntentionResult, String>;

    /// Resolve using heuristic keyword matching (existing Planner logic).
    fn resolve_heuristic(&self, text: &str) -> IntentionResult;
}
```

**AI Resolution Flow:**
1. Build a system prompt that describes the available capabilities (Capability enum), action types, and the Goal struct schema
2. Send user text to AI via `AIEngine::chat()` or `AIEngine::complete()`
3. Parse the AI response into a structured `Goal`
4. If AI returns a clarification question → return `ClarificationNeeded`
5. If AI returns multiple sub-goals → return `Goals(vec![...])`
6. If AI fails or is unavailable → fall back to heuristic resolver

**Tool-based resolution (preferred):**
The AI has a `resolve_intention` tool with JSON-schema parameters:
```json
{
  "name": "resolve_intention",
  "description": "Resolve a user's natural language request into an actionable goal",
  "parameters": {
    "type": "object",
    "properties": {
      "description": { "type": "string", "description": "Goal description" },
      "context": { "type": "object", "description": "Optional context parameters" },
      "needs_clarification": { "type": "boolean" },
      "clarification_question": { "type": "string" },
      "sub_goals": {
        "type": "array",
        "items": { "type": "object", "properties": {
          "description": { "type": "string" },
          "context": { "type": "object" }
        }}
      }
    },
    "required": ["description"]
  }
}
```

**Heuristic Fallback:**
Wraps the existing `Planner::decompose()` logic. If the heuristic produces steps, wrap them in a `Goal`. If not, return `Unsupported`.

**SQLite Schema:**
```sql
CREATE TABLE IF NOT EXISTS goals (
    id TEXT PRIMARY KEY,
    original_text TEXT NOT NULL,
    resolved_goal TEXT NOT NULL,        -- JSON
    status TEXT NOT NULL DEFAULT 'submitted',
    plan_id TEXT,
    report_id TEXT,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    completed_at INTEGER
);

CREATE TABLE IF NOT EXISTS reports (
    id TEXT PRIMARY KEY,
    goal_id TEXT NOT NULL,
    plan_id TEXT NOT NULL,
    report_json TEXT NOT NULL,
    summary TEXT,
    created_at INTEGER NOT NULL,
    FOREIGN KEY (goal_id) REFERENCES goals(id)

CREATE INDEX idx_goals_status ON goals(status);
CREATE INDEX idx_goals_created ON goals(created_at);
CREATE INDEX idx_reports_goal ON reports(goal_id);

-- Full-text search for goal text
```

### Subsystem 3 — Execution Manager

**File:** `execution_manager.rs`

**Purpose:** Central orchestrator for the goal lifecycle. Manages submission, AI resolution, planning, execution (via PlanExecutor), persistence, and queueing.

```rust
/// Status of a goal in the execution lifecycle.
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
pub enum GoalLifecycleStatus {
    Submitted,              // Goal submitted, awaiting resolution
    Resolving,              // NL→Goal resolution in progress
    Resolved,               // Goal resolved, awaiting consent
    PendingConsent,         // Waiting for user consent (via event)
    Planned,                // Plan ready, awaiting execution
    Executing,              // Execution in progress
    Completed,              // All steps completed
    Failed(String),         // Execution failed
    Cancelled,              // User cancelled
    Interrupted,            // App restarted during execution
}

/// A goal in the execution queue.
pub struct GoalEntry {
    pub id: String,
    pub original_text: String,
    pub status: GoalLifecycleStatus,
    pub goal: Option<Goal>,
    pub plan: Option<ExecutionPlan>,
    pub report: Option<GoalExecutionReport>,
    pub summary: Option<String>,
    pub created_at: i64,
    pub updated_at: i64,
}

/// Progress information for a running execution.
pub struct ExecutionProgress {
    pub goal_id: String,
    pub total_steps: usize,
    pub completed_steps: usize,
    pub current_step: Option<usize>,
    pub current_description: Option<String>,
    pub elapsed_ms: i64,
}

/// Manages the full lifecycle of goal execution.
pub struct ExecutionManager {
    intention_parser: Arc<IntentionParser>,
    goal_registry: Arc<GoalRegistry>,
    plan_executor: Arc<PlanExecutor>,
    planner: Arc<Planner>,
    world_state: Arc<RwLock<WorldState>>,
    event_bus: Option<tokio::sync::broadcast::Sender<NovaEvent>>,
    config: ExecutionManagerConfig,

    // Internal state
    queue: Arc<parking_lot::Mutex<VecDeque<GoalEntry>>>,
    active: Arc<AtomicBool>,
    active_goal_id: Arc<parking_lot::Mutex<Option<String>>>,
}

impl ExecutionManager {
    pub fn new(
        intention_parser: Arc<IntentionParser>,
        goal_registry: Arc<GoalRegistry>,
        plan_executor: Arc<PlanExecutor>,
        planner: Arc<Planner>,
        world_state: Arc<RwLock<WorldState>>,
        config: ExecutionManagerConfig,
    ) -> Self;

    pub fn with_event_bus(mut self, bus: tokio::sync::broadcast::Sender<NovaEvent>) -> Self;

    /// Submit a natural language goal for execution.
    pub async fn submit(&self, text: &str) -> Result<String, AutomationError>;
    /// Submit a pre-built Goal struct for execution.
    pub async fn submit_goal(&self, goal: Goal, text: Option<&str>) -> Result<String, AutomationError>;

    /// Get the current status of a goal.
    pub fn status(&self, goal_id: &str) -> Result<Option<GoalEntry>, AutomationError>;
    /// List goals with optional filter.
    pub fn list(&self, filter: GoalFilter) -> Result<Vec<GoalEntry>, AutomationError>;
    /// Get execution progress of the active goal.
    pub fn progress(&self) -> Option<ExecutionProgress>;

    /// Cancel a goal (queued or executing).
    pub fn cancel(&self, goal_id: &str) -> Result<(), AutomationError>;
    /// Cancel all queued goals.
    pub fn cancel_all(&self) -> Result<u64, AutomationError>;

    /// Resume an interrupted goal.
    pub fn resume(&self, goal_id: &str) -> Result<(), AutomationError>;

    /// Process the next queued goal (called internally by the execution loop).
    async fn process_queue(&self);
}
```

**Lifecycle Flow:**

```
submit(text)
  │
  ├─→ Save GoalRecord (status: Submitted)
  ├─→ Publish GoalSubmitted event
  ├─→ Enqueue GoalEntry
  │
  ▼
process_queue()
  │
  ├─→ IntentionParser::resolve(text) → Goal(s)
  │     ├─→ Success → update status: Resolved
  │     ├─→ Clarification → publish GoalNeedsClarification event, wait
  │     └─→ Unsupported → publish GoalUnsupported event, fail
  │
  ├─→ ConsentGate::check_plan(plan) → pre-classify stakes
  │     ├─→ Auto-approved (low stakes, autonomous mode) → proceed
  │     └─→ Needs consent → publish GoalPendingConsent event, wait
  │
  ├─→ Planner::plan(goal) → ExecutionPlan
  │     └─→ update status: Planned
  │
  ├─→ PlanExecutor::execute_plan(plan) → GoalExecutionReport
  │     └─→ update status: Executing
  │
  ├─→ Save report, generate summary
  ├─→ Publish GoalCompleted / GoalFailed event
  └─→ Process next in queue
```

**Persistence during lifecycle:**
- At every lifecycle transition, the `GoalRegistry` is updated
- On restart: load all goals with status `Executing` or `Submitted` → mark `Interrupted`
- `resume()` re-starts an interrupted goal's plan from the last completed step

## Risks

| ID | Risk | Mitigation | Likelihood | Impact |
|----|------|------------|------------|--------|
| R1 | AI hallucinates goals that don't match user intent | Always show plan summary before execution (consent gate pre-classification). Add confidence threshold — low confidence → ask user to confirm. | Medium | High |
| R2 | AI resolution introduces latency (3-10s per goal) | Heuristic fallback for simple/known patterns. Parallel resolution of sub-goals. Configurable timeout. | Medium | Medium |
| R3 | SQLite contention between GoalRegistry and other subsystems | Use separate DB file (not shared with memory/search). WAL mode for concurrent access. Connection pooling via r2d2. | Low | Medium |
| R4 | Restart during execution leaves inconsistent state | Transactional status updates. Mark interrupted on startup. Provide resume() method. | Low | High |
| R5 | AI consumes too many tools, generating many automation goals | Rate limiting on AI tool calls. Max concurrent goals = 1 (sequential queue). Configurable max goals per conversation. | Low | Medium |
| R6 | Ambiguous goals cause infinite clarification loop | Max clarification rounds (configurable). After limit, fall back to heuristic or return Unsupported. | Low | Low |
| R7 | GoalRegistry FTS query performance degrades | Indexed query alternatives. Configurable max_records with auto-purge. | Low | Low |
