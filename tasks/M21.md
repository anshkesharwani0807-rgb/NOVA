# M21 — Closed-Loop Autonomous Execution

> **Status:** IMPLEMENTED — COMPLETE
> **Depends on:** M20 (Planner + World State) — COMPLETE
> **Frozen milestones:** M17 (nova_screen), M18 (nova_input), M19 (Task Execution), M20 (Planning + World State)
> **Location:** `modules/automation/src/` (new files) + minor additions to existing files
> **Key constraint:** No changes to any frozen public API. No breaking changes. No new modules.

---

## Objective

Connect the M20 Planner, M20 World State, M19 Execution Engine, M19 ComputerController, M17 nova_screen, and M18 nova_input into a **closed-loop autonomous execution pipeline**. The system must be able to: accept a user goal → gather device/world context → produce a plan → execute plan steps with screen-aware feedback → verify each step's outcome → recover from failures → report completion.

This is the missing integration layer that transforms independent subsystems into an autonomous agent.

---

## Why M21 Logically Follows M20

| Milestone | What it built | Role |
|-----------|---------------|------|
| M17 | nova_screen: capture, OCR, UI tree, visual grounding | **Sense** - perceive screen state |
| M18 | nova_input: mouse/keyboard/touch simulation | **Act** - interact with UI |
| M19 | ComputerController, real executors, consent gate, retry | **Act** - task execution with recovery |
| M20 S1 | Planner: goal decomposition, topological sort, plans | **Plan** - decide what to do |
| M20 S2 | WorldState: device telemetry, diffs, subscriptions | **Context** - know current state |
| **M21** | **Connect Plan → Act → Sense → Context into closed loop** | **Autonomous Agent** |

M21 does not build new fundamental capabilities. It **integrates** the five existing subsystems into a pipeline where:
- Planner output feeds Execution Engine input
- WorldState informs both planning and execution
- Screen feedback verifies execution outcomes
- Failures trigger re-planning from the failed step

## Functional Requirements

### FR-M21-001 — Goal-to-Execution Pipeline
The system MUST accept a `Goal`, gather current context from `WorldState`, produce an `ExecutionPlan` via the `Planner`, convert the plan into executable units, and feed them to the `ExecutionEngine` for sequential execution — all in one orchestrated call.

### FR-M21-002 — WorldState-Aware Step Execution
Before executing each step, the system MUST consult `WorldState` to check preconditions. For example: if a step sets Wi-Fi, skip it if already in desired state. If a step opens an app, verify it isn't already in the foreground.

### FR-M21-003 — Screen-Verified Outcome Detection
After executing a screen-interaction step (click, type, drag, swipe), the system MUST capture a new frame, run OCR/grounding, and verify the expected outcome occurred. For non-screen steps (device control), consult `WorldState.verify_change()`.

### FR-M21-004 — Adaptive Recovery on Failure
If a step fails verification, the system MUST attempt recovery: retry with backoff (reuse M19 retry logic), then re-plan from the failed step with updated WorldState context, then escalate to the user with a clear explanation.

### FR-M21-005 — Plan Progress Tracking
The system MUST track which steps of a `ExecutionPlan` are completed, in-progress, failed, or pending. This state MUST be queryable during execution and persisted in execution history.

### FR-M21-006 — Event Bus Integration for Pipeline Events
Every stage of the pipeline (plan created, step started, step verified, step failed, plan completed, plan abandoned) MUST publish typed `AutomationEventPayload` variants on the kernel event bus.

### FR-M21-007 — WorldState Snapshot at Plan Start
At the start of a goal execution, the system MUST capture a `WorldSnapshot` that serves as the baseline context for the entire plan. Steps can reference this snapshot to detect environmental changes.

### FR-M21-008 — Consent Gate Integration
Every step in the pipeline MUST pass through the consent gate (reuse M19 `ConsentGate`). The pipeline pre-classifies the entire plan at creation time, showing the user a summary of stakes before execution begins ("This plan will: click 3 buttons, type 2 fields — medium stakes").

## Internal Subsystems (Dependency-Ordered)

### Subsystem 1 — Pipeline Step & Plan Adapter

**Files:** `pipeline_step.rs`, `execution_plan_adapter.rs`

**`PipelineStep`** struct:

```rust
pub struct PipelineStep {
    pub step: ExecutionStep,            // from the planner
    pub step_index: usize,
    pub status: PipelineStepStatus,
    pub preconditions: Vec<Precondition>,
    pub expected_outcome: ExpectedOutcome,
    pub recovery_strategy: RecoveryStrategy,
    pub pre_execution_snapshot: Option<WorldSnapshot>,
    pub post_execution_snapshot: Option<WorldSnapshot>,
}

pub enum PipelineStepStatus {
    Pending, InProgress, Succeeded, Failed(String), Skipped(String),
}

pub enum Precondition {
    DeviceState { field: String, expected: String },
    AppNotRunning(String),
    AppIsRunning(String),
    ScreenContains(String),   // OCR text visible
    NetworkState { online: Option<bool>, wifi: Option<bool> },
}

pub enum ExpectedOutcome {
    ScreenChange { description: String },
    DeviceStateChange { field: String },
    AppForeground { app_name: String },
    TextEntered { target: String, text: String },
    NoChange,
}

pub enum RecoveryStrategy {
    Retry(u32),                    // retry N times with backoff
    ReplanFromThisStep,            // call Planner::replan from this step
    ReplanFromStart,               // call Planner::replan entirely
    Skip,                          // skip and continue
    Abort,                         // stop the plan
    EscalateToUser,                // ask user what to do
}
```

**`ExecutionPlanAdapter`** — converts `ExecutionPlan` → `Vec<PipelineStep>`:

```rust
pub struct ExecutionPlanAdapter;

impl ExecutionPlanAdapter {
    pub fn convert(
        plan: &ExecutionPlan,
        world_snapshot: Option<&WorldSnapshot>,
    ) -> Vec<PipelineStep>;

    pub fn derive_preconditions(
        step: &ExecutionStep,
        world: Option<&WorldSnapshot>,
    ) -> Vec<Precondition>;

    pub fn derive_expected_outcome(step: &ExecutionStep) -> ExpectedOutcome;

    pub fn derive_recovery_strategy(step: &ExecutionStep) -> RecoveryStrategy;
}
```

`derive_preconditions` uses heuristic logic:
- `DeviceControl(SetBrightness)` → check current brightness from WorldSnapshot
- `DeviceControl(ToggleWiFi(true))` → skip if wifi already enabled
- `ClickScreenElement` → `ScreenContains` precondition
- `OpenApp` → `AppNotRunning` precondition

`derive_expected_outcome` maps action types to verifiable outcomes:
- `ClickScreenElement` → `ScreenChange { "new screen after click" }`
- `TypeIntoScreenElement` → `TextEntered { target, text }`
- `DeviceControl(SetBrightness)` → `DeviceStateChange { "brightness" }`
- `OpenApp` → `AppForeground { app_name }`

### Subsystem 2 — Outcome Verifier

**File:** `outcome_verifier.rs`

```rust
pub struct OutcomeVerifier {
    world_state: Arc<RwLock<WorldState>>,
    screen: Option<Arc<RwLock<ScreenEngine>>>,
}

impl OutcomeVerifier {
    pub fn new(
        world_state: Arc<RwLock<WorldState>>,
        screen: Option<Arc<RwLock<ScreenEngine>>>,
    ) -> Self;

    /// Verify the outcome of a completed step.
    pub async fn verify(
        &self,
        step: &PipelineStep,
        action_result: &ActionResult,
    ) -> VerificationResult;

    /// Verify device state change against WorldState.
    pub async fn verify_device_state(
        &self,
        field: &str,
        expected: &str,
        prev_snapshot: Option<&WorldSnapshot>,
    ) -> VerificationResult;

    /// Capture screen and check if expected text/state is present.
    pub async fn verify_screen_state(
        &self,
        expected: &str,
    ) -> VerificationResult;
}

pub enum VerificationResult {
    Passed,
    Failed { reason: String, suggestion: String },
    Uncertain { reason: String },
}
```

Logic:
- For `DeviceStateChange`: capture new `DeviceTelemetry` via `DeviceTelemetryCollector`, compare with previous snapshot
- For `ScreenChange`: capture frame, re-run OCR, diff UI tree, check if substantial change occurred
- For `AppForeground`: capture frame, run OCR for app name, check grounding
- For `TextEntered`: capture frame, run OCR near expected element location, verify text present
- For `NoChange`: always returns `Passed`
- Uses `WorldState.compute_diff()` between pre- and post-execution snapshots

### Subsystem 3 — Recovery Orchestrator

**File:** `recovery_orchestrator.rs`

```rust
pub struct RecoveryOrchestrator {
    planner: Arc<Planner>,
    execution_engine: Arc<ExecutionEngine>,
    outcome_verifier: Arc<OutcomeVerifier>,
    world_state: Arc<RwLock<WorldState>>,
    config: AutomationConfig,
}

impl RecoveryOrchestrator {
    pub fn new(
        planner: Arc<Planner>,
        execution_engine: Arc<ExecutionEngine>,
        outcome_verifier: Arc<OutcomeVerifier>,
        world_state: Arc<RwLock<WorldState>>,
        config: AutomationConfig,
    ) -> Self;

    /// Handle a failed step according to its recovery strategy.
    pub async fn handle(
        &self,
        step: &PipelineStep,
        plan: &ExecutionPlan,
        context: &HashMap<String, String>,
        publish: &dyn Fn(AutomationEventPayload),
    ) -> RecoveryOutcome;
}

pub enum RecoveryOutcome {
    Retried(PipelineStep),          // step executed again, may have succeeded
    Replanned(ExecutionPlan),       // new plan from fault point
    Skipped,                        // step skipped, continue rest of plan
    Aborted(String),                // plan abandoned
    Escalated { reason: String },   // sent to user, awaiting input
}
```

Recovery decision tree:
1. If `RecoveryStrategy::Retry(n)` and retries < n → retry with exponential backoff
2. If retries exhausted and `RecoveryStrategy::ReplanFromThisStep` → call `Planner::replan(plan, failed_index, current_snapshot)` to produce a new plan from the failure point
3. If `RecoveryStrategy::ReplanFromStart` → call `Planner::plan()` with original goal + current WorldSnapshot
4. If `RecoveryStrategy::Skip` → mark step as `Skipped`, continue
5. If `RecoveryStrategy::Abort` → stop entire plan, publish failure
6. If `RecoveryStrategy::EscalateToUser` → publish `PlanNeedsInput` event, wait for user response via `WorldSubscription`-like callback

### Subsystem 4 — Plan Executor (Main Orchestrator)

**File:** `plan_executor.rs`

```rust
pub struct PlanExecutor {
    planner: Arc<Planner>,
    world_state: Arc<RwLock<WorldState>>,
    execution_engine: Arc<ExecutionEngine>,
    outcome_verifier: Arc<OutcomeVerifier>,
    recovery_orchestrator: Arc<RecoveryOrchestrator>,
    adapter: ExecutionPlanAdapter,
    event_bus: Option<tokio::sync::broadcast::Sender<NovaEvent>>,
}

impl PlanExecutor {
    pub fn new(
        planner: Arc<Planner>,
        world_state: Arc<RwLock<WorldState>>,
        execution_engine: Arc<ExecutionEngine>,
        config: AutomationConfig,
    ) -> Self;

    pub fn with_screen(mut self, screen: Arc<RwLock<ScreenEngine>>) -> Self;
    pub fn with_event_bus(mut self, bus: tokio::sync::broadcast::Sender<NovaEvent>) -> Self;

    /// Execute a goal through the full pipeline.
    pub async fn execute_goal(
        &self,
        goal: Goal,
    ) -> Result<GoalExecutionReport, AutomationError>;

    /// Get current execution status.
    pub fn status(&self) -> Vec<PipelineStepStatus>;

    /// Cancel a running execution.
    pub fn cancel(&self, execution_id: &str);
}

pub struct GoalExecutionReport {
    pub goal: Goal,
    pub plan: ExecutionPlan,
    pub pipeline_steps: Vec<PipelineStep>,
    pub overall_status: GoalStatus,
    pub started_at: i64,
    pub completed_at: Option<i64>,
    pub total_duration_ms: i64,
    pub verification_results: Vec<VerificationResult>,
}

pub enum GoalStatus {
    Completed,
    Partial(Vec<String>),       // list of skipped/failed steps
    Failed(String),
    Cancelled,
    Escalated { reason: String },
}
```

**`execute_goal` flow:**

```
1. Capture WorldSnapshot → baseline context
2. Publish PlanExecutionStarted event
3. Call Planner::plan(goal_with_context) → ExecutionPlan
4. Publish PlanCreated event (includes step summary)
5. Call ExecutionPlanAdapter::convert(plan, snapshot) → Vec<PipelineStep>
6. For each PipelineStep (indexed, sequential):
   a. Mark PipelineStepStatus::InProgress
   b. Publish StepStarted event
   c. Check preconditions via WorldState current state → if unmet, skip/mark
   d. Execute via ExecutionEngine (reuses existing execute_step_direct with consent gate)
   e. Capture post-execution WorldSnapshot
   f. Call OutcomeVerifier::verify(step, action_result) → VerificationResult
   g. If verification Failed:
      - Publish StepFailed event
      - Call RecoveryOrchestrator::handle() → RecoveryOutcome
      - Apply recovery outcome (retry/replan/skip/abort/escalate)
   h. If verification Passed:
      - Mark PipelineStepStatus::Succeeded
      - Publish StepCompleted event
   i. If cancelled → break
7. Build GoalExecutionReport
8. Publish PlanCompleted / PlanFailed / PlanAborted event
9. Return report
```

## Risks

| ID | Risk | Mitigation | Likelihood | Impact |
|----|------|------------|------------|--------|
| R1 | Pipeline too slow for real-time use (verification frame capture dominates latency) | Add skip-verification mode for time-sensitive goals; make verification configurable per-step | Medium | Medium |
| R2 | Re-planning from failed step produces same plan (infinite loop) | Add plan fingerprint + dedup: if re-plan produces identical steps, escalate to user | Low | High |
| R3 | Screen-based verification is fragile (different apps, dynamic content) | Default to `Uncertain` (not `Failed`) when confidence is borderline; log for human review | Medium | Medium |
| R4 | ExecutionPlanAdapter precondition/outcome heuristics miss edge cases | Make `derive_preconditions` and `derive_expected_outcome` extensible (trait or function registry) for future AI-driven derivation | Low | Low |
| R5 | Pipeline introduces circular dependency between subsystems | PlanExecutor orchestrates by composing existing independent modules; no circular Cargo deps | Low | High |
| R6 | Race conditions: WorldState changes between verification and execution | Capture snapshot at start of each step, use it for verification, not live state | Low | Medium |
